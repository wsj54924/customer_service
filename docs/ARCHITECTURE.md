## 三、技术选型 (8GB VRAM 适配)

### 方案: 云端API + 本地Embedding/Reranker

| 模块 | 技术 | 理由 |
|------|------|------|
| **Web框架** | FastAPI | 异步高性能，自动生成API文档 |
| **文本Embedding** | BAAI/bge-large-zh-v1.5 (本地) | 中文SOTA, ~1.3GB, 8GB显存可跑 |
| **Reranker** | BAAI/bge-reranker-v2-m3 (本地) | 中英双语cross-encoder |
| **向量存储** | FAISS (CPU) | 轻量高效，无需额外服务 |
| **文本生成** | Qwen-Plus (API) | 中文最强，成本低 |
| **多模态理解** | Qwen-VL-Max (API) | 原生多模态，理解图片 |
| **备选API** | OpenAI GPT-4o/mini | 英文题目fallback |
| **文档解析** | 自研parser | 保留`<PIC>`和图片ID映射 |

### 为什么不本地跑大模型?
- 8GB VRAM只能跑4bit量化的7B模型，效果差
- API模型(Qwen-Plus/VL)效果远超本地小模型
- 400题的API成本约几元人民币

## 四、核心模块设计

### 4.1 文档解析 (ManualParser)
- 解析20份中文手册 + 1份英文汇总手册
- 按`<PIC>`边界分割，保留图片ID映射
- 按chunk_size=512字符切分，overlap=64

### 4.2 向量检索 (FAISS + BGE)
- 所有chunks用bge-large-zh-v1.5编码为1024维向量
- FAISS IndexFlatIP索引，余弦相似度
- 检索top-10 → rerank取top-5

### 4.3 问题分类 (Query Router)
- 产品使用类 → 从手册检索，需返回图片
- 客服服务类 → 通用客服话术，无需图片
- 混合类 → 两种策略结合

### 4.4 答案生成 (LLM)
- 构建prompt: system指令 + RAG上下文 + 用户问题
- 产品问题: 使用vision模型理解图片+手册内容
- 服务问题: 使用text模型基于通用规则回答

### 4.5 多轮对话 (Session Memory)
- 内存字典存储每个session的对话历史
- 保留最近3轮对话上下文
- session_id由客户端传入或自动生成

## 八、风险与备选

| 风险 | 影响 | 应对 |
|------|------|------|
| API调用延迟 | 批量答题慢 | 并发+缓存常见问答 |
| BGE模型显存不足 | 无法本地部署 | 用更小的embedding模型或API |
| 图片ID匹配错误 | 丢分 | 精细化匹配规则+人工校验 |
| 英文手册理解差 | 英文题丢分 | 用GPT-4o处理英文题 |
